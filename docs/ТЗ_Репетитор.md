# Детальные ТЗ по функциональностям (Шаг 4.3)

## F1. Парсинг и управление контентом ФИПИ

**Цель:** Наполнить базу заданий из открытого банка ФИПИ для предметов "История" и "Обществознание".

* **Вход:** URL-адреса страниц банка ФИПИ или ручной ввод администратором/репетитором.

* **Выход:** Заполненные сущности `subjects`, `themes`, `tasks` в БД.

* **Валидация:**

* Обязательные поля: `text_content`, `type` (TEST/ESSAY), привязка к `theme_id`.

* Для TEST: наличие `correct_answer_key`.

* Для ESSAY: наличие `fipi_criteria`.

* **Хранение медиа-файлов:**

* Изображения заданий загружаются на S3 при парсинге/импорте.

* В `tasks.text_content` хранятся URL-ссылки на S3 в формате JSON: `{"text": "...", "images": ["https://s3.../img1.jpg", "https://s3.../img2.jpg"]}`.

* При ошибке загрузки изображение помечается как `broken` в metadata задания.

* **Форматы JSON для критериев и ключей:**

* **Пример `correct_answer_key` для TEST-заданий:**

```json

{

"type": "short_answer",

"correct_answer": "Москва"

}

```

или

```json

{

"type": "multiple_choice",

"correct_answer": [1, 3, 5]

}

```

* **Пример `fipi_criteria` для ESSAY-заданий:**

```json

[

{"id": "criterion_1", "name": "Названа причина", "max_score": 1},

{"id": "criterion_2", "name": "Приведён пример", "max_score": 2},

{"id": "criterion_3", "name": "Использованы термины", "max_score": 1}

]

```

* **Edge-кейсы и ошибки:**

* *Изменение верстки ФИПИ:* Парсер падает. Решение: при ошибке парсинга задание не сохраняется, администратор получает алерт, задание можно добавить вручную через UI.

* *Дубликаты:* ФИПИ может содержать одинаковые задания в разных разделах. Решение: проверка на дубликат по хэшу `text_content`.

* *Отсутствие изображений:* В задании есть ссылка на картинку, которая недоступна. Решение: скачивать картинку на наш сервер (S3) при парсинге, при ошибке — помечать задание как `broken`.

* **Критерии приёмки:**

1. Система успешно парсит и сохраняет минимум 100 заданий по Истории и 100 по Обществознанию с корректной привязкой к дереву тем.

2. При ручном добавлении задания администратором система не дает сохранить его без указания типа и привязки к теме.

3. При попытке добавить задание с идентичным текстом система выдает ошибку "Задание уже существует".

## F2. Регистрация, роли и связи (IAM)

**Цель:** Обеспечить легальную регистрацию пользователей с учетом 152-ФЗ и настроить связи между ролями.

* **Вход:** Email, пароль, ФИО, дата рождения, роль (TUTOR/STUDENT/PARENT), чекбоксы согласий.

* **Выход:** Созданные записи в `users`, `profiles`, `tutor_student`, `student_parent`.

* **Валидация:**

* Email уникален. Пароль соответствует политике безопасности.

* Дата рождения валидна (не в будущем, не старше 100 лет).

* **Ограничения:**

* Один email = одна учётная запись = одна роль (TUTOR, STUDENT или PARENT). Пользователь не может иметь несколько ролей одновременно.

* Регистрация доступна всем пользователям независимо от возраста. Дата рождения сохраняется для статистики, но не влияет на доступ к функционалу.

* **Edge-кейсы и ошибки:**

* *Отказ в согласии на ПДн:* Регистрация невозможна.

* **Критерии приёмки:**

1. Пользователь не может зарегистрироваться, не проставив чекбокс согласия на обработку ПДн.

2. Репетитор может сгенерировать "код приглашения", по которому ученик при регистрации автоматически попадает в связь `tutor_student`.

## F3. Конструктор тестов и назначения

**Цель:** Дать репетитору инструмент для сборки теста из базы и назначения его ученикам.

* **Вход:** Выбор предмета, тем (кодификатор), типов заданий (TEST/ESSAY), лимит времени, массив `student_ids`.

* **Выход:** Сущности `tests`, `test_tasks`, `test_assignments`, уведомления ученикам.

* **Логика генерации:**

* При создании теста репетитор выбирает темы и количество заданий. Система случайным образом выбирает задания из выбранных тем и сохраняет фиксированный набор в `test_tasks`. Все ученики, которым назначен этот тест, получают одинаковый набор заданий.

* **Валидация:**

* Лимит времени — опциональное поле. Может быть `NULL` (без лимита) или `> 0`. Если `NULL`, таймер не отображается ученику.

* Выбранные темы существуют и содержат задания нужного типа.

* **Логика назначений (test_assignments):**

* При вызове `POST /tests/{id}/assign` бэкенд создаёт записи в таблице `test_assignments` для каждого `student_id` со статусом `ASSIGNED`.

* Когда ученик начинает тест (создаётся `attempt`), статус `test_assignments` обновляется на `IN_PROGRESS`.

* Когда `attempt` становится `COMPLETED`, статус `test_assignments` обновляется на `COMPLETED`.

* Для ретестов создаются новые записи в `attempts`, но `test_assignments` остаётся `COMPLETED` после первой сдачи.

* **Edge-кейсы и ошибки:**

* *Недостаточно заданий:* Репетитор выбрал тему, но в ней всего 2 задания TEST, а он запросил 5. Решение: UI показывает доступное количество и не дает сохранить тест, либо автоматически ограничивает количество.

* *Ученик удален/отвязан:* Репетитор пытается назначить тест ученику, связь с которым уже разорвана. Решение: Валидация на уровне API, возврат ошибки `STUDENT_NOT_LINKED`.

* **Критерии приёмки:**

1. Репетитор может собрать тест, выбрав 3 темы из кодификатора, и система сгенерирует тест из случайных заданий этих тем (набор фиксируется).

2. При назначении теста массиву `student_ids` каждому ученику становится доступна ссылка на прохождение, и создаются записи в `test_assignments`.

3. Репетитор не может сохранить тест, если в выбранных темах нет заданий указанного типа.

## F4. Движок прохождения теста (с автосохранением)

**Цель:** Обеспечить бесшовное прохождение теста учеником с защитой от потери данных и списывания.

* **Вход:** ID теста, действия ученика (старт, ввод ответа, завершение).

* **Выход:** Сущности `attempts`, `answers`.

* **Валидация:**

* Статус попытки: `IN_PROGRESS`.

* Таймер: отсчет времени ведется **строго на сервере** (backend-driven). Клиент только отображает оставшееся время.

* **Edge-кейсы и ошибки:**

* *Обрыв связи:* Ученик пишет ответ, интернет пропал. Решение: Фронтенд сохраняет ответ в `IndexedDB`. При восстановлении связи отправляет `PATCH` с текущим содержимым. Бэкенд применяет стратегию **Last-Write-Wins**: принимает последнюю полученную версию без сравнения timestamp'ов.

* *Закрытие вкладки:* Ученик закрыл вкладку. Таймер на сервере продолжает идти. При повторном входе ученик видит оставшееся время.

* *Истечение таймера:* Время вышло. Бэкенд возвращает `400 ATTEMPT_CLOSED`. Фронтенд блокирует UI и редиректит на страницу результатов. Фоновая задача (Scheduler) каждые 60 секунд проверяет просроченные попытки и автоматически меняет их статус на `PENDING_REVIEW`/`COMPLETED` с запуском автопроверки.

* **Критерии приёмки:**

1. Если ученик обновит страницу или закроет вкладку, все введенные до этого ответы сохраняются и отображаются при повторном входе.

2. Если ученик попытается изменить системное время на ПК, таймер на сервере не изменится, и тест завершится по серверному времени.

3. При истечении времени бэкенд возвращает `400 ATTEMPT_CLOSED`, фронтенд блокирует UI, а фоновая задача автоматически завершает попытку.

## F5. Автопроверка тестовой части

**Цель:** Мгновенно проверить задания типа TEST по ключам ФИПИ.

* **Вход:** Завершенная попытка (`Attempt`), ответы ученика на задания типа TEST.

* **Выход:** Обновленный `auto_score` в сущности `Attempt` и `Answer`.

* **Типы заданий и форматы ключей:**

| Тип задания | Формат JSON ключа | Алгоритм сверки |

|-------------|-------------------|-----------------|

| Краткий ответ (слово/число/дата) | `"correct_answer": "Москва"` или `"correct_answer": 1945` | Приведение к нижнему регистру, удаление лишних пробелов, точное совпадение |

| Выбор одного варианта | `"correct_answer": 3` | Точное совпадение номера |

| Выбор нескольких из N | `"correct_answer": [1, 3, 5]` | Проверка на вхождение всех элементов (порядок не важен) |

| Соответствие (два списка) | `"correct_answer": {"1": "А", "2": "В", "3": "Б"}` | Проверка всех пар (ключ-значение) |

| Последовательность (установить порядок) | `"correct_answer": [2, 4, 1, 3]` | Точное совпадение порядка |

* **Логика смены статуса:**

* Если в тесте есть хотя бы одно ESSAY-задание → статус `PENDING_REVIEW`.

* Если в тесте только TEST-задания → статус сразу `COMPLETED` после автопроверки.

* **Синхронность выполнения:**

* Автопроверка TEST-заданий выполняется синхронно в рамках запроса `POST /submit`.

* Если время обработки превышает 5 сек, рассмотреть перенос в фоновую задачу с поллингом статуса.

* **Edge-кейсы и ошибки:**

* *Опечатки и регистр:* Ученик написал "москва" вместо "Москва". Решение: Приведение к нижнему регистру и удаление лишних пробелов перед сверкой.

* *Несколько вариантов ответа:* В ключе может быть несколько верных вариантов (например, "1, 2" или "2, 1"). Решение: Парсинг ключа в массив и проверка на вхождение всех элементов.

* **Критерии приёмки:**

1. При сдаче теста система мгновенно вычисляет балл за тестовую часть и возвращает его ученику.

2. Система корректно засчитывает ответ, написанный в нижнем регистре или с лишними пробелами, если по смыслу он верен (согласно ключу).

3. Задания типа ESSAY не участвуют в расчете `auto_score`.

## F6. Проверка развернутых ответов с AI-подсказкой

**Цель:** Дать репетитору инструмент для быстрой ручной проверки с помощью AI-анализа.

* **Вход:** `task_id`, `student_answer`.

* **Выход:** `ai_feedback` (текст разбора), `suggested_scores` (предложенные баллы по критериям).

* **Валидация:**

* Статус ответа: ожидает проверки.

* Для MVP запросы к GigaChat выполняются **синхронно** с HTTP-таймаутом 30 сек. Асинхронная очередь (Celery) с rate-limiting отложена на пост-MVP.

* **Edge-кейсы и ошибки:**

* *Ошибка/Таймаут GigaChat:* API недоступен или превышен таймаут 30 сек. Решение: В UI отображается статус "AI временно недоступен", репетитор может проверить ответ без подсказки.

* *Галлюцинации AI:* AI предложил 0 баллов за верный ответ. Решение: В UI кнопки "Применить оценку AI" **нет**. Репетитор обязан вручную выставить баллы по каждому критерию, AI-подсказка носит только информационный характер.

* **Критерии приёмки:**

1. При нажатии "Запросить AI-подсказку" система возвращает текстовый разбор соответствия ответа критериям ФИПИ (синхронно, в рамках HTTP-запроса).

2. Репетитор не может завершить проверку задания, не проставив вручную баллы по каждому критерию из `fipi_criteria`.

3. После проверки всех развернутых заданий в попытке статус `Attempt` автоматически меняется на `COMPLETED`, и ученику/родителю становятся доступны итоговые баллы.

## F7. Аналитика и Дашборды

**Цель:** Визуализировать прогресс учеников для репетитора, ученика и родителя.

* **Вход:** ID пользователя (репетитор/ученик/родитель).

* **Выход:** Агрегированные данные (динамика баллов, % правильных ответов по темам).

* **Расчёт метрик по темам:**

* Для TEST-заданий: `% правильных = (количество правильных ответов / общее количество TEST-заданий) * 100%`.

* Для ESSAY-заданий: `% выполнения = (сумма manual_score / сумма max_score) * 100%`.

* Общая метрика по теме: средневзвешенное значение % выполнения по всем заданиям темы (TEST + ESSAY).

* **Логика агрегации ретестов:**

* Все попытки ученика группируются по `theme_id` (через связь `task → theme`).

* Для каждой темы рассчитываются: тренд (первая попытка vs последняя), средний балл по всем попыткам, количество попыток.

* **Роли и доступ:**

* **Роль "Родитель" НЕ включена в MVP.**.

* **Edge-кейсы и ошибки:**

* *Ретесты:* Ученик пересдал тему. Решение: В аналитике отображаются **все** попытки. Базовая метрика — средний % или тренд (последняя попытка vs первая).

* *Нет данных:* Ученик еще не проходил тесты. Решение: Отображается заглушка "Пока нет данных".

* **Критерии приёмки:**

1. Репетитор видит список всех своих учеников и может отфильтровать их по "слабым темам" (где % правильных ответов < 50%).

2. В профиле ученика отображается график динамики баллов по датам (учитываются все ретесты).  
  
🛠️ Техническая декомпозиция и подводные камни (Для бэкенда и фронта)

Критерий 1: Фильтр по «слабым темам» (< 50%) у Репетитора

Как это будет выглядеть в коде:

Репетитор открывает список учеников. Бэкенд должен для каждого ученика посчитать процент правильных ответов по всем темам и пометить те, где < 50%.

🔴 Риск производительности (N+1 и тяжелые JOIN): Если у репетитора 50 учеников, и каждый сдал по 10 тестов, то при каждом открытии списка бэкенд будет делать тяжелый GROUP BY по таблицам attempts -> answers -> tasks -> themes. На MVP (один репетитор, пара десятков учеников) PostgreSQL это «проглотит», но запрос будет выполняться 1-2 секунды.

💡 Решение для MVP: Делаем «ленивую» агрегацию. Считаем метрики только для тех учеников, которые уже прошли хотя бы один тест.

⚠️ Важное уточнение для бэкенда: В расчёт success_rate (успешности по теме) должны попадать только попытки в статусе COMPLETED. Если попытка в статусе PENDING_REVIEW (развернутые ответы еще не проверены репетитором), её ESSAY-задания не должны участвовать в знаменателе, иначе процент будет некорректным.

Критерий 2: График динамики баллов (с учетом ретестов)

Как это будет выглядеть в коде:

Фронтенд рендерит график (например, на Recharts или Chart.js), где ось X — это даты, ось Y — баллы.

🔴 Edge-кейс «Несколько попыток в один день»: Ученик в субботу прошел тест по теме «Петр I» (получил 40%), расстроился, пересдал его в тот же день (получил 80%).

Вопрос к фронту/бэкенду: Как рисуем точку на графике за субботу?

💡 Решение для MVP: Бэкенд в эндпоинте GET /analytics/dashboard должен группировать попытки не просто по дате, а агрегировать их. Например, возвращать max_score за день или average_score за день.

Моя рекомендация: Возвращаем максимальный балл за дату (это лучше мотивирует ученика и показывает его «потолок» на текущий момент). Формат ответа должен быть строго: [{"date": "2026-07-01", "score": 80}, ...].

⚠️ Edge-кейс «Пустые данные»: Если ученик только что зарегистрировался, API возвращает dynamics: []. Фронтенд обязан отрендерить красивый Empty State («Пока нет данных», как указано в ТЗ F7), а не сломанный график с пустыми осями.
